# Отчёт по РГЗ: Видеоплатформа VideoHub

## 1. Общее описание проекта

В рамках РГЗ разработана полноценная веб-платформа для потокового видео — **VideoHub**. Платформа позволяет пользователям регистрироваться, загружать видео, просматривать контент других пользователей, ставить лайки и оставлять комментарии.

---

## 2. Используемые технологии и их обоснование

### Бэкенд — Django REST Framework
- **Django 4.2** выбран за зрелость экосистемы, встроенную ORM и удобную систему миграций.
- **DRF (Django REST Framework)** обеспечивает быстрое создание REST API с сериализацией, авторизацией и пагинацией из коробки.
- **JWT-аутентификация** (библиотека `djangorestframework-simplejwt`) — стандартный подход для SPA: access-токен хранится в localStorage, автоматически обновляется через refresh-токен.

### База данных — PostgreSQL
Выбран PostgreSQL за надёжность, поддержку сложных запросов и хорошую интеграцию с Django. Для локальной разработки подходит SQLite, но для production — только PostgreSQL.

### Фронтенд — React 18
- **React** с хуками (`useState`, `useEffect`, `useContext`) — современный подход без классовых компонентов.
- **React Router v6** для клиентской маршрутизации (SPA).
- **Axios** с перехватчиками (interceptors) для автоматического добавления JWT-токена к каждому запросу и обновления токена при 401-ошибке.

---

## 3. Архитектурные решения

### Структура бэкенда
Приложение разбито на два Django-приложения:
- `users` — регистрация, авторизация, профили
- `videos` — CRUD видео, лайки, комментарии, категории

Такое разделение следует принципу **Single Responsibility** (из SOLID).

### Кастомная модель пользователя
Использована кастомная модель `User` (наследует `AbstractUser`) с полем `email` в качестве логина. Это рекомендованный Django-паттерн: менять модель после миграций крайне сложно.

### Паттерн DRY в сериализаторах
`VideoListSerializer` содержит базовые поля. `VideoDetailSerializer` наследует его и добавляет `video_url` и `comments`. Это исключает дублирование кода.

### Безопасность
- Пароли хранятся в хэшированном виде (Django PBKDF2).
- CORS настроен только на `localhost:3000`.
- Изменять/удалять видео может только автор (кастомный permission `IsAuthorOrReadOnly`).
- Счётчик просмотров обновляется атомарно (`F('views_count') + 1`), без гонки потоков.

## 5. Трудности и пути решения

### Трудность 1: Загрузка больших файлов
**Проблема:** При загрузке видео через стандартный `axios.post` запрос может долго не отвечать.  
**Решение:** Добавлен прогресс-бар и сообщение «Загрузка...». В production рекомендуется использовать Nginx с увеличенным `client_max_body_size` и отдельный media-сервер.

### Трудность 2: Гонка потоков при обновлении счётчика просмотров
**Проблема:** `video.views_count += 1; video.save()` небезопасно при параллельных запросах.  
**Решение:** Использование `F`-выражений: `Video.objects.filter(pk=pk).update(views_count=F('views_count') + 1)` — атомарная операция на уровне БД.

### Трудность 3: Автоматическое обновление JWT
**Проблема:** Access-токен истекает через 1 час, пользователь не должен перелогиниваться.  
**Решение:** В axios-interceptor при получении 401 автоматически делается запрос на `/api/auth/token/refresh/` и повторяется исходный запрос с новым токеном.

### Трудность 4: CORS при разработке
**Проблема:** React (порт 3000) обращается к Django (порт 8000) — браузер блокирует запросы.  
**Решение:** Библиотека `django-cors-headers` + настройка `CORS_ALLOWED_ORIGINS`.

## 6. Принципы разработки

- **DRY** — базовые сериализаторы наследуются, CSS-переменные вместо дублирования стилей
- **SOLID** — разделение на приложения (`users`, `videos`), кастомные permissions
- **Модульность** — каждый React-компонент в отдельном файле, сервисный слой (`api.js`)
- **REST** — правильные HTTP-методы (GET/POST/PATCH/DELETE) и статус-коды
